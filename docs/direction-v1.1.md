# 产品方向调整 v1.1 — 从「任务工具」到「目标推进闭环」

> 状态：方向草案 | 基于与产品负责人的需求讨论收敛而成
> 配套：PRD v1.0、mvp-scope.md、project-modes-design.md（本文档是后者的上位收敛）

## 0. 为什么调整

MVP v1.0 验证的是核心循环「说一句话 → 拆解 → 排期 → 完成」。实际使用暴露出**项目 tab 鸡肋**，根因经讨论澄清为五条真实痛点：

| 痛点（用户原话） | 真实需求 | v1.0 现状 |
|---|---|---|
| 想到什么能快速记到待办 | 零摩擦捕捉 | ✅ 已满足（快速添加） |
| 项目多不好管理 | 多目标的组织 | ⚠️ 半残：无新建入口、归类靠 AI 猜 |
| **没有推进感** | 看见自己在前进 | ❌ 最痛：假进度条帮倒忙 |
| **没有复盘迭代** | 回看 + 调整 | ❌ 完全没有：完成即消失 |
| 记录想法让 AI 辅助管理目标 | AI 当管家 | ⚠️ 只用于拆任务，未用于管目标 |

**关键洞察**：项目「鸡肋」不是因为项目无用，而是因为用**单一进度条**去量化所有项目。持续运营型项目（小红书/视频号）没有终点，进度条只会「越投入越倒退」，制造焦虑。解法不是删项目，而是**换掉量化方式 + 补上推进感与复盘**。

---

## 1. 目标形态：四块闭环

```
捕捉(两个平等入口) → 今日清单(唯一执行场) → 推进感(项目) → 复盘
                          ↑__________________________|
```

- **捕捉**：快速记 与 项目驱动 两个入口平等，但汇入同一个今日清单。
- **执行**：今日清单是唯一执行场（排期 → 专注 → 完成）。
- **推进感**：完成的任务回流到项目，项目用「自定义标准」量化，显示势头而非进度条。
- **复盘**：展示已有但从未呈现的数据（完成记录、跳过归因、耗时偏差）。

---

## 2. 捕捉：两个平等入口，一个归宿

**原则：两个入口平等，但任务最终都汇入同一个今日清单。用户永远不被强迫「先建项目」。**

| 入口 | 流程 | 适合 |
|------|------|------|
| 快速记（保留现状） | 一句话 → 今日清单（默认归「随手记」，不问项目） | 突发、急活、零碎想法 |
| 项目驱动（新增） | 进项目 → 加任务 → 同步到今日清单（带项目归属与标准） | 围绕明确目标规划的事 |

**避免的坑**：不能做成「必须先选项目再加任务」——那会退化成传统任务工具（PRD 第一条痛点：用户花大量时间管理工具）。快速记必须始终是零摩擦的默认可达入口。

**设计要点**：
- 两个入口都终结于今日清单这一唯一执行场，不存在「项目里的任务」和「清单里的任务」两套对不上的体系。
- 项目驱动入口加的任务，可标「每日重复」（见 §3 的日更标准），自动每天进清单。

---

## 3. 项目：用户自定义标准，量化「势头」而非「完成度」

**核心突破**：系统算不出持续型项目的进度，但**用户能自己定标准**。把「成功」的定义权交给用户，量化难题即化解。

沿用 project-modes-design.md 的三态，但强化「用户定标准」：

| 模式 | 用户填的标准 | 显示的推进感 | 会倒退？ |
|------|------------|-------------|---------|
| `count` 有限计数 | 目标件数（读 100 本书） | 圆环 = 完成数/目标数，可结业变金色 | 否，只涨 |
| `streak` 持续投入 | **每日/每周标准**（日更 1 个视频） | 连续 N 天达标 · 本周 X/7 达标 · 累计 M 件 | 不设总进度，断更只归零不报错 |
| `result` 结果目标 | 目标数值（涨粉到 1 万） | 数值条 = 当前/目标，「记一笔」手动更新，按周期结算 | 周期重置 |

**关键改动**：
- 持续型项目（streak）**不显示进度条**，改显示「连续达标天数 + 本周达标 + 累计件数」——这就是用户要的「推进感」，且因标准是用户自定义的真实承诺，不再是假指标。
- count 圆环分母从 `total_tasks` 改为 `goal_target`，解决「越加任务越倒退」。
- 模式由用户建项目时手动选，**不调 AI**（低频、强主观）。AI 只用在快速记时归类到已有项目（高频、客观）。
- 「随手记」固定 count 且无目标，是收件箱，不参与结业。

---

## 4. 复盘：把已有数据展示出来

**关键事实**：隐形学习（profile_learn）已经在记录复盘所需的全部数据——完成时长、实际vs预估耗时偏差、跳过归因——但**从未展示给用户**。复盘页几乎是「把现有数据呈现出来」，不需要新埋点。

复盘页（新增，建议放周维度）展示：
- **本周做了啥**：完成任务数、按项目分布（哪个目标推进最多）
- **哪些在拖**：被反复跳过的任务 + 归因（「没状态」偏多=精力错配；「等待外部」偏多=被阻塞）
- **耗时认知**：实际 vs 预估偏差（duration_bias），让用户看见自己习惯性高估/低估多少
- 不做评判性文案，只陈述事实 + 轻量建议（延续 coach.ts 的支持性口吻）

---

## 5. 数据结构变更（相对现有）

`projects` 集合新增（来自 project-modes-design.md）：

| 字段 | 类型 | 说明 |
|------|------|------|
| mode | string | `count`/`streak`/`result`，默认 `count` |
| goal_target | number? | count=目标件数；result=目标数值；streak 用 daily_quota |
| daily_quota | number? | streak 的每日标准（如日更 1 个），新增 |
| goal_unit | string? | result 单位（元/粉丝） |
| cycle | string? | result 结算周期 month/week/none |
| current_value | number? | result 当前值，手动记 |
| current_value_at | number? | result 最近更新时间 |

`tasks` 已具备 `parent_task_id`/`scheduled_date`，项目驱动的「每日重复任务」需新增：

| 字段 | 类型 | 说明 |
|------|------|------|
| repeat | string? | `daily`/`none`，每日重复任务每天自动进清单 |

## 6. 落地顺序（建议，可分期）

**第一期（解决最痛的「推进感」）**：
1. 后端 project_create/project_list 收发 mode/goal/daily_quota 字段
2. 前端项目页加「＋ 新建项目」入口 + 模式选择面板
3. projects.ts 按 mode 分流渲染：count 圆环修分母、streak 显示连续天数/本周达标/累计

**第二期（补「复盘」）**：
4. 新增复盘页，展示完成分布 / 跳过归因 / 耗时偏差（数据已有）

**第三期（完善项目驱动）**：
5. 项目内加任务 + 一键同步今日清单
6. 每日重复任务（repeat=daily）自动进清单
7. result 模式「记一笔」+ 周期重置

**原则**：每期独立可用、可验证，不必一次做完。第一期就能让项目 tab 从「鸡肋」变「有推进感」。

---

## 7. 不做什么（守住边界）

- ❌ 不做幕布式无限文档/嵌套大纲——会把执行工具变成笔记工具，想法进了文档库就不再被执行，违背「专注当下」内核。要笔记直接用幕布。
- ❌ 不强制「先建项目再加任务」——保住快速记的零摩擦。
- ❌ 持续型项目不做假进度条。
- ✅ 守住独有能力：把一句话变成排好期的行动。这是别的工具没有的，是产品的命根子。
